# Scenario: classification — Traffic Type Classification

> **Status: May contain bugs**
> This scenario was developed before several backend and frontend changes. Known issues and potential bugs are listed in the [Known Issues](#known-issues) section.

## Overview

`classification` is a single-agent scenario where the agent must classify network traffic into one of four types — **NONE, PING, UDP, TCP** — based on global packet and byte counters observed from the Mininet OVS switch.

Unlike the attack detection scenarios, there is no adversarial component and no network-level response action (no link blocking). The agent purely observes and classifies.

---

## Network Topology

Same star topology as other scenarios (one OVS switch, N hosts, OpenDaylight controller). Traffic is generated by the scenario engine using `ping`, `iperf UDP`, or `iperf TCP` flows; the agent observes aggregate switch port statistics.

---

## Working Mode

Two operating modes controlled by `gym_type`:

- **`CLASSIFICATION`** — live Mininet traffic; the environment generates traffic in real time and the agent observes OVS counters.
- **`CLASSIFICATION_FROM_DATASET`** — replays a pre-recorded traffic trace from file. Episode count is derived automatically from the file length.

Three phases:

1. **Training** — agents train for their computed episode count; optional per-agent chart output.
2. **Evaluation** — agents are tested on a held-out set; accuracy, precision, recall, F1 computed.
3. **Plotting** — confusion matrices, training curves, network status plots saved to the experiment directory.

---

## Observation Space

**Space type:** `Box(shape=(4,), dtype=float64)`

| Index | Feature | Bounds |
|-------|---------|--------|
| 0 | Received packets | `[0, threshold_packets]` |
| 1 | Transmitted packets | `[0, threshold_packets]` |
| 2 | Received bytes | `[0, threshold_bytes]` |
| 3 | Transmitted bytes | `[0, threshold_bytes]` |

**Default thresholds** (`scenario_env_param.yaml`):

```yaml
classification:
  thresholds:
    packets: 200
    bytes: 1000000     # 1 MB
```

Note: unlike the attack scenarios the observation is **global** (aggregated across all hosts), not per-host. There are no delta/percentage-change features; only absolute counters are used.

**Normalisation:** SB3 agents receive a normalised state (`get_normalized_state()`). Tabular agents call `get_discretized_state()` which uses logarithmic binning (`np.logspace`) per dimension.

---

## Action Space

**Space type:** `Discrete(4)`

| Action | ID | Traffic type |
|--------|----|-------------|
| `NONE` | 0 | No traffic |
| `PING` | 1 | ICMP ping traffic |
| `UDP` | 2 | UDP iperf traffic |
| `TCP` | 3 | TCP iperf traffic |

The action is a pure classification label; no network reconfiguration is performed.

---

## Reward Function

The reward is **graded by distance** from the correct answer, assuming the action IDs have a meaningful ordinal structure (NONE < PING < UDP < TCP):

| Condition | Reward |
|-----------|--------|
| Correct classification (`action == correct`) | +1.0 |
| Off by one (`|action − correct| = 1`) | −0.80 |
| Off by two (`|action − correct| = 2`) | −0.90 |
| Completely wrong (`|action − correct| = 3`) | −1.0 |

The rationale is to give partial negative feedback when the agent is "close" (e.g. confusing UDP with TCP) vs. completely off (e.g. classifying heavy iperf as NONE).

> **Note:** The ordinal distance assumption only makes sense if traffic types are ordered by intensity/similarity. The current ID ordering (NONE=0, PING=1, UDP=2, TCP=3) partially reflects this, but the grading is not semantically rigorous.

## State Representation

The `InstantState` object stores global network totals:

```python
{
    received_packets:    int,
    transmitted_packets: int,
    received_bytes:      int,
    transmitted_bytes:   int,
    total_packets:       int,   # cumulative across episode
    total_bytes:         int,
    packets_percentage_change: float,
    bytes_percentage_change:   float,
}
```

`get_state()` returns the 4-element list `[received_packets, transmitted_packets, received_bytes, transmitted_bytes]` used as the observation.

Per-host states are also maintained (8-feature arrays as in other scenarios) but are **not exposed in the observation** — the agent only sees the aggregate.

**Ground truth** is the `generated_traffic_type` integer (0–3), set by the scenario engine before each step based on what traffic is currently being generated.

## Configuration Parameters

```yaml
classification:
  thresholds:
    packets: 200
    bytes: 1000000
    var_packets: 100    # defined but not used
    var_bytes: 100      # defined but not used
```

The `var_packets` and `var_bytes` fields are present in the config but are never read by the environment (see [Known Issues](#known-issues)).

## Known Issues

The following bugs and inconsistencies were identified. They may need fixing as the backend/frontend evolves:

1. **Unused variance thresholds** — `var_packets` and `var_bytes` are defined in `scenario_env_param.yaml` but never read by `NetworkEnvClassification`. The observation space has no delta features, making these parameters dead configuration.

2. **Commented-out original observation space** — two lines in `network_env_classification.py` show a previous 4-feature definition that included variance terms (`var_packets`, `var_bytes`). The current observation uses only absolute counters. This is either a deliberate simplification or an incomplete migration.

3. **`if` instead of `elif` in `execute_action()`** — the action dispatch uses separate `if` statements rather than `if/elif`. If an action value is somehow invalid, the variable `msg` may be unset, causing a `NameError`. Should be an `if/elif/else` chain.

4. **`state.setter` bug in base `InstantState`** — the property setter references the variable `status` which is not in scope; it should reference `value`. This may cause a `NameError` at runtime when the setter is triggered.

5. **Discretization offset** — `get_discretized_state()` adds 1 to each raw value before calling `np.digitize` (`val + 1`). The reason for this offset is not documented and may introduce boundary errors for values near 0.

6. **Sleep proportional to traffic type ID** — in `update_state()`, the environment sleeps for `generated_traffic_type + 0.5` seconds after generating traffic. This means TCP (ID=3) always waits 3.5 s while NONE (ID=0) waits only 0.5 s. This hardcoded coupling between class ID and wait time is fragile.

7. **Episode count truncation from dataset** — when running in `CLASSIFICATION_FROM_DATASET` mode, episodes are computed as `int((len(statuses) - test_episodes) / (max_steps + 1))`. Integer division silently discards the remainder, potentially leaving unused data.

8. **Evaluation summary exception silently swallowed** — failed `notify_client()` calls during evaluation summary are caught and logged only at DEBUG level, meaning failures are invisible in normal operation.
