# Scenario: attacks — Binary Attack Detection (Network-Level)

> **Status: May contain bugs**
> This scenario predates several backend and frontend additions. Known inconsistencies and bugs are listed in the [Known Issues](#known-issues) section.

## Overview

`attack_detect` is the simplest attack detection scenario: a **single agent** observes **aggregate network-level** traffic statistics and must decide whether the network is currently under attack or not. The decision is binary (NORMAL vs. ATTACK) and is taken at the level of the whole network, not per host.

This is the conceptual predecessor of `attack_ho`. The key difference is granularity: here the agent sees a 4-dimensional global summary and produces one binary label per step, whereas `attack_ho` sees 8 per-host features and classifies each host independently.

---

## Network Topology

Standard star topology: one OVS switch, N hosts, OpenDaylight SDN controller. Traffic is generated by a background adversarial process (in live mode) or replayed from a dataset file.

---

## Working Mode

Two operating modes, controlled by `gym_type`:

- **`ATTACKS`** — live Mininet network; traffic is generated continuously by an adversarial agent; the environment reads OVS counters every step.
- **`ATTACKS_FROM_DATASET`** — replays a pre-recorded trace from file; episode count is derived from file length.

Three phases:

1. **Training** — agents train sequentially or in live mode; optional per-agent charts.
2. **Evaluation** — agents tested on held-out episodes; accuracy, precision, recall, F1 computed.
3. **Plotting** — environment execution statuses saved to file and visualised.

---

## Observation Space

**Space type:** `Box(shape=(4,), dtype=float32)`

| Index | Feature | Bounds |
|-------|---------|--------|
| 0 | Total packets (all hosts) | `[0, threshold_packets × num_hosts]` |
| 1 | Δ packets (% change) | `[-threshold_var_packets, +2 × threshold_var_packets]` |
| 2 | Total bytes (all hosts) | `[0, threshold_bytes × num_hosts]` |
| 3 | Δ bytes (% change) | `[-threshold_var_bytes, +2 × threshold_var_bytes]` |

**Default thresholds** (`scenario_env_param.yaml`):

```yaml
attacks:
  thresholds:
    packets: 22000
    var_packets: 50
    bytes: 423000000
    var_bytes: 30
```

> Note: the upper bound for delta features is `2 × threshold_var` (asymmetric), unlike `attack_ho` where it is symmetric. This was likely an early design choice and may be revisited.

**Normalisation:** SB3 agents receive normalised observations via `get_normalized_state()`. Tabular agents use `get_discretized_state()`.

---

## Action Space

**Space type:** `Discrete(2)`

| Action | ID | Meaning |
|--------|----|---------|
| `NORMAL_TRAFFIC` | 0 | Network traffic is normal |
| `ATTACK` | 1 | Network is under attack |

No network reconfiguration is performed. The action is a pure classification label — unlike `attack_ho`, there is no SDN link blocking.

---

## Reward Function

| Condition | Reward |
|-----------|--------|
| Correct attack detection | +1.0 |
| Correct normal classification | +1.0 |
| False negative (missed attack) | −2.0 |
| False positive (false alarm) | −1.0 |

False negatives are penalised more heavily than false positives, prioritising recall over precision.

> **Note:** The constant `CORRECT_ATTACK_DETECTION = 2.0` is defined in `constants.py` but is **not used** in `calculate_reward()`, which hardcodes `1.0` for correct attack detection. See [Known Issues](#known-issues).

---

## State Representation

The `InstantState` holds global network metrics:

```python
status = {
    "id":     int,       # 0: normal, 1: attack  (-1: idle)
    "status": str,       # "normal" | "attack"
    "packets": int,
    "bytes":   int,
    "packets_percentage_change": float,
    "bytes_percentage_change":   float,
}
```

Per-host arrays (8 features each, same structure as `attack_ho`) are also maintained internally and used to derive the global status: if any host is classified as `under_attack` or `attacking`, the global status becomes `attack`.

**Status ID mappings:**

| ID | Global status |
|----|--------------|
| −1 | idle |
| 0 | normal |
| 1 | attack |

| ID | Host status |
|----|------------|
| −1 | idle |
| 0 | normal |
| 1 | under\_attack |
| 2 | attacking |

---

## Configuration Parameters

```yaml
attacks:
  likely: 0.45          # Attack probability per step
  apply_drop_rules: false
  thresholds:
    packets: 22000
    var_packets: 50
    bytes: 423000000
    var_bytes: 30
```

This scenario's config is simpler than `attack_ho`: there are no `likely_train`/`likely_eval` split parameters, no attack duration or unblock policy settings.

---

## Comparison with attack\_ho

| Aspect | attack\_detect | attack\_ho |
|--------|---------------|------------|
| Observation | 4-dim global aggregate | 8-dim per-host slice |
| Action | Discrete(2): Normal / Attack | Discrete(3): Normal / Attack\_In / Attack\_Out |
| Granularity | Whole network | Per host |
| SDN response | None | Link blocking via drop rule |
| Wrapper | None | `PerHostScanWrapper` (micro-step decomposition) |
| Attack direction | Not distinguished | Victim vs. attacker distinguished |
| Reward asymmetry | FN > FP | Configurable scale; direction error penalised separately |

## Known Issues

The following bugs and inconsistencies were identified:

1. **Reward constant mismatch** — `CORRECT_ATTACK_DETECTION = 2.0` is defined in `constants.py` but `calculate_reward()` hardcodes `return 1.0` for correct attack detection. The constant is never used, making it misleading.

2. **`raise` with a string argument** — in `evaluate_attack_detect_agent()`, invalid model handling uses `raise("The model can't be None. ...")`. In Python 3 this raises a `TypeError` rather than the intended error. Should be `raise ValueError(...)`.

3. **`if` instead of `elif` in `execute_action()`** — action dispatch uses separate `if` statements. If `action` is somehow invalid, `msg` is never assigned and the function raises `UnboundLocalError`. Should be `if/elif/else`.

4. **Asymmetric delta bounds** — observation indices 1 and 3 (percentage changes) have `high = 2 × threshold_var`, giving an asymmetric range `[-var, +2×var]`. This differs from `attack_ho` where the range is symmetric `[-var, +var]`. The asymmetry is likely unintentional and may confuse normalisation.

5. **`IDLE` action defined but unused** — `AGENT_ACTIONS.IDLE = -1` is defined in `constants.py`. The commented-out guard `if self.generated_traffic_type == AGENT_ACTIONS.IDLE: return 0.0` was removed but the constant remains. The action space has no idle action (only 0 and 1).

6. **`TODO` in `evaluate_traffic()`** — the state array is constructed manually each call rather than as a property of `InstantState`. This is a known refactor target flagged with a TODO comment.

7. **No attack duration / streak parameters** — unlike `attack_ho`, this scenario has no `likely_train`, `likely_eval`, `short_attack_duration`, `long_attack_duration`, or unblock policy parameters. If these are needed in future they must be added to `scenario_env_param.yaml` and wired into the environment.

8. **Live mode step delay** — in `ATTACKS` mode, `step()` sleeps 1 second after each non-terminal step (`time.sleep(1)`) to allow the network to generate traffic. This sleep is hardcoded and not configurable via params.
